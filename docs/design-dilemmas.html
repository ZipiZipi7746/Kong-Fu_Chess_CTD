The Extension Requirement

“The system must remain highly extensible. In the future, the board may not be rectangular, piece movement rules may be completely different from standard chess, and game variants should be introduced with minimal modification to existing stable code.”

This changes how every existing TODO in this codebase should be read. Rectangular rows/cols, standard chess pieces, delta-based movement, capture-by-overwrite, and king-capture victory are implementations of the current iteration, not permanent engine assumptions. Open/Closed is therefore a primary project requirement, not an optional future concern — and every dilemma below is re-argued against that standard, not against “is this convenient for the current chess-only game.”

♟ Four forks in the road

Each one is a real trade-off, not a right-answer-vs-wrong-answer. The verdict names the cost of the option that won, not just its benefit — and, where the extension requirement changes the calculus, says so explicitly.

Strategy Pattern · piece_rules.py

The Pawn problem

Every other piece answers legality with four integers: matches(dr, dc). Pawn can't — it needs color, absolute position, and what's sitting on the board. Widen the interface everyone shares, or give the one piece that needs more its own door?

✕ Considered

Widen matches() for all six

def matches(self, piece,
    from_row, from_col,
    to_row, to_col, board):
    ...

One uniform hook, textbook Strategy. But King, Rook, Bishop, Queen and Knight have never needed board or piece — they'd carry parameters they immediately ignore, and five already-green test classes would need rewriting for zero behavior change.

✓ Chosen

A Template Method default, Pawn overrides it

class PieceRule:
    def is_legal(self, piece,
        from_row, from_col,
        to_row, to_col, board):
        dr = abs(to_row - from_row)
        dc = abs(to_col - from_col)
        return self.matches(dr, dc)

class PawnRule(PieceRule):
    def is_legal(self, ...):
        ...  # full context, own rules

RuleEngine calls rule.is_legal(...) uniformly for all six kinds — no piece.kind == "P" branch survives. The five simple rules, and every one of their existing tests, are untouched byte-for-byte.

Verdict

A pragmatic current win, not the final contract. Chose blast radius over purity: PawnRule inherits a matches() it never implements, and is_legal's default still measures dr/dc — a rectangular-grid delta, exactly as grid-specific as before. Under the extension requirement, that's now a named gap rather than a footnote: a genuinely topology-independent RuleContext (piece, positions, board, topology, game state, clock) is still required, staged deliberately after Position/BoardTopology settle rather than guessed at here.

Dependency Injection · gui/rendering/renderer.py

One interface, one legal implementation

Renderer is an ABC with exactly one production implementation, by explicit instructor constraint — “ImgRenderer is the only implementation that exists or is allowed.” If nothing else may ever stand behind it, is the interface pulling any weight, or is it ceremony?

✕ Considered

Drop the abstraction

# game_loop.py, calling
# ImgRenderer directly
renderer = ImgRenderer(content)
renderer.draw_sprite(...)

Honest about the constraint: there will only ever be one backend, so stop pretending otherwise. Fewer moving parts, one less file.

✓ Chosen

Keep it — for a different reason than usual

# Tests/gui/animation/
#   test_view_model_registry.py
class FakeRenderer:
    def draw_sprite(...): ...

DI has two separate payoffs: swappable production backends (explicitly off the table here) and swappable test doubles. game_loop and ViewModelRegistry depend on Renderer, not on cv2 — and a FakeRenderer already exercises that seam in the test suite.

Verdict

The two payoffs don't have to travel together, and a non-rectangular board doesn't reopen this question either way: an irregular or hex board still paints through ImgRenderer, at pixel rects Renderer never has an opinion about. What changes is which coordinates get handed to it — that's a future BoardLayout's job (see Dilemma 4), not a second Renderer backend. Declined a “swap rendering backends” TODO here on purpose; the testability benefit already in use is reason enough on its own.

Encapsulation & Strategy · model/board.py

How far does one pass go?

Last cycle's fix was narrow: BoardRenderer reaching into board.cells directly, a two-line encapsulation fix. This cycle, the explicit non-rectangular-board requirement turned one item from that same file's TODO list — boundary validity — from “someday” into “now.” Which one gets a real seam this time, and which stay on the list?

✕ Considered

Fix everything board.py's TODOs name

# in one pass:
# - full BoardTopology
#   (positions/neighbors)
# - BoardStorage
# - Position value object
# - validated get_cell/set_cell

They're all named in the same file, all pointing at the same rectangular assumption. Batching them means one review instead of four — but a Position value object alone touches roughly thirteen files across every layer.

✓ Chosen

One minimal, provably real seam

class RectangularTopology:
    def contains(self, row, col):
        return (0 <= row < self.rows
            and 0 <= col < self.cols)

# Board(cells, topology=None)
# is_inside() -> self.topology
#     .contains(row, col)

Board.is_inside now asks an injected object, defaulting to today's rectangle — zero callers changed, behavior identical, and a test proves it by injecting a topology a rectangle would reject.

Verdict

A seam is not the same thing as the abstraction. This is contains() only, still keyed by (row, col) — not positions()/neighbors(), not a topology-independent Position, not migrated into RuleEngine or the GUI. Each of those remains a named, required-direction TODO (not speculative — the requirement asks for exactly this), deferred because each has a different, larger blast radius than a single default-parameter injection. Related in the write-up still isn't related in the risk.

Staged Extensibility · engine/game_engine.py

General engine, or chess-specific engine?

GameEngine still knows what a King is, still resolves every arrival as source-to-destination travel, still decides victory inline. A truly variant-agnostic engine would know none of that. Generalize the whole orchestration layer now, or extract one seam at a time?

✕ Considered

Generalize everything in one pass

# in one rewrite:
# - GameAction (Command)
# - InteractionResolver
# - full BoardTopology
# - WinCondition
# all wired through GameEngine
# simultaneously

Would produce the “properly generic” engine the requirement ultimately points at — but as one large, hard-to-review rewrite touching every method GameEngine has, with no test coverage proving any single seam correct before the next one lands on top of it.

✓ Chosen

Extract the one seam that was safe to isolate

class CaptureKingWinCondition:
    def check(self, moving_piece,
        captured_piece, board):
        if (captured_piece and
            captured_piece.is_king()):
            return moving_piece.color
        return None

# GameEngine(..., win_condition=None)

WinCondition was genuinely isolated (one inline if, one clear question: “did this arrival just end the game”) — extracted, injected, tested with a fake that proves the engine no longer hardcodes King capture. InteractionResolver and GameAction were judged not isolated enough yet and stayed TODOs.

Verdict

OCP is being reached incrementally, by identifying real extension seams and implementing the ones that are already isolated - not by declaring one large framework necessary and building it speculatively. That is slower to look “done,” and is the honest trade-off: a reviewer should see exactly which seams exist today, which were added this pass, and which are still staged — not a claim that the engine is already variant-agnostic.

How to present this in the review

  • Name the assumptions as implementations, not contracts, up front. Rectangular geometry, chess pieces, capture-by-overwrite, and king-capture victory are all things the current code does, not things the architecture requires.
  • Point at the seams that already exist before this pass: Strategy-pattern rules, DI throughout GameEngine/RuleEngine/GameController, a Renderer interface already exercised by a fake.
  • Show which seams were added now — BoardTopology, WinCondition, the shared piece registry — and that each has a test proving it's genuinely injectable, not just structurally present.
  • Be explicit about what crosses too many layers to do safely in this pass — Position, full BoardTopology, BoardLayout, InteractionResolver — and why: each touches multiple files with no way to verify correctness incrementally.
  • Deferral is not the same as ignoring the requirement. Every deferred item is named, classified (required-direction vs. feature-triggered vs. rejected), and reasoned about in this document and in the code comment at its exact seam.
  • Overengineering all of this in one commit would have cost more than it bought — harder to review, harder to verify, and more likely to guess wrong about interfaces no second implementation has tested yet. Staged extensibility is the deliberate choice, not a shortcut.

Suggested order for the review

  1. The Extension Requirement, read aloud — frame everything that follows against it before showing any code.
  2. Dilemma 4 (general vs. chess-specific engine) — the clearest statement of the staged-extensibility strategy; start here, not with Dilemma 1.
  3. Run the suite on screen — pytest Tests -v, 406 green, same two pre-existing unrelated subprocess failures as before this pass.
  4. The extension map and OCP table — let the reviewer see the honest weak spots before they ask about them.
  5. Dilemmas 1–3 — each is a smaller, concrete instance of the same staging decision.
  6. TODO status groups — confirm every deferral reads as reasoned, not as an excuse.
